## 2. Benefits & Challenges

### Benefits

**Speed of iteration.** Features that would have taken me hours to research and prototype were available within minutes as a starting point. This let me focus more time on understanding, testing, and refining rather than writing from scratch.

**Bilingual implementation.** The Arabic/English RTL feature involved CSS direction flipping, dynamic DOM updates, and localStorage persistence simultaneously. Claude helped me see how these three concerns could be coordinated cleanly without adding a third-party i18n library.

**Accessibility awareness.** Claude consistently included ARIA attributes (`aria-label`, `aria-expanded`, `aria-valuenow`) in generated code, which reinforced good practices I had learned in class.

**Documentation quality.** Claude helped structure the README and technical docs in a professional format, which I then customised with accurate details about my own project.

### Challenges

**Over-generation.** Claude sometimes generated more code than needed — for example, it initially included a lo-fi music player which I decided was not professional enough for a software engineering portfolio. I had to explicitly ask for alternatives and then remove the generated code.

**Context drift across sessions.** Because Claude does not retain memory between conversations, I had to re-upload my files at the start of each session. This required me to keep track of which version was the latest.

**RTL edge cases.** Some CSS RTL overrides generated by Claude did not behave correctly in Safari — specifically, the `flex-direction: row-reverse` on the navbar conflicted with the mobile hamburger behaviour. I debugged and fixed this manually.

**Placeholder text for Arabic.** Claude initially missed adding `data-ar-placeholder` attributes to the message textarea. I caught this during testing and asked for a correction.

## 3. Learning Outcomes

**Bidirectional text (BiDi) and RTL layout.** This project gave me hands-on experience with CSS `direction`, `dir` HTML attributes, and the challenges of designing interfaces that must work in both LTR and RTL reading modes. I learned that flipping a layout is not just about `direction: rtl` — it requires careful attention to padding, borders, flex order, and text alignment throughout.

**Data attributes as a translation mechanism.** Using `data-en` and `data-ar` directly on HTML elements is a simple and effective pattern for small-scale internationalisation without a library. I now understand when this approach is appropriate and when a proper i18n library like `i18next` would be better.

**AI as a starting point, not a solution.** Every piece of AI-generated code required review, testing, and often modification. I learned to treat AI output as a first draft — useful for structure and boilerplate, but not production-ready without my own understanding and corrections.

**Scroll events and performance.** Implementing the scroll progress bar reinforced the importance of `{ passive: true }` on scroll listeners and debouncing for performance on low-end devices.

**Accessibility in practice.** Seeing ARIA attributes in context — such as `aria-expanded` on the mobile nav toggle and `aria-valuenow` on the progress bar — helped me understand how screen readers interact with dynamic content.
